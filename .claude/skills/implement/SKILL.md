# バグ修正・機能追加の実装ワークフロー

## ステップ 1: mainからブランチを切る

```bash
git checkout main
git pull origin main
git checkout -b <branch-name>
```

**ブランチ名のルール**（ユーザーが指定した場合はそれを優先）:
- バグ修正: `fix/<内容を表す短い英語スラッグ>`
- 機能追加: `feat/<内容を表す短い英語スラッグ>`
- 改善・リファクタ・パフォーマンス: `chore/<内容を表す短い英語スラッグ>`
- スラッグはケバブケース（例: `feat/add-task-filter`）

> 要件が曖昧でブランチ名が決まらない場合は、先にステップ 2 の確認を済ませてからこのステップに戻り、ブランチを切って実装に進む。

## ステップ 2: 不明点があれば質問する

実装に着手する前に、要望の中で曖昧・不明な点があれば AskUserQuestion ツールで確認する。

**質問すべき例**:
- 挙動の詳細が仕様として定まっていない
- 複数の実装アプローチがあり、どちらを優先するか判断できない
- 影響範囲が広く、どこまで修正するか不明

**質問不要な例**:
- 要望が明確で実装方針が自明
- コードを読めば意図が汲める

## ステップ 3: 実装する

通常どおり実装を行う。動作確認を依頼する前にコミットする。

```bash
git add <files>
git commit -m "<conventional commit message>"
```

## ステップ 4: 動作確認をユーザーに依頼する

実装が完了したら、ユーザーに動作確認を依頼する。具体的に何を確認してほしいかを伝える。

例:
```
実装が完了しました。以下を確認していただけますか？

- 〇〇の動作: ～～が期待通りに動くか
- エッジケース: ～～の場合に問題がないか

問題なければ PR を作成します。
```

## ステップ 5: OKをもらったらPRを作成する

ユーザーが動作確認して OK を出したら、main に向けて PR を作成する。

```bash
git push -u origin <branch-name>
gh pr create --base main --title "<タイトル>" --body "$(cat <<'EOF'
## Summary
- <変更内容の箇条書き>

## Test plan
- [ ] <確認項目>

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

PR URL をユーザーに返す。
